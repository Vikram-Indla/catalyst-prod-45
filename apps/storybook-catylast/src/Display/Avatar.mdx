<Subtitle>
  The visual identity primitive — paired with a name in comments,
  mention popovers, table row-actions, profile menus, and assignee
  pickers. Renders a profile photo when one is available and falls back
  to a colored initials tile with a deterministic, name-hashed
  background.
</Subtitle>

`Avatar` ships from **`@catylast/primitives`**. Its job is small
on the surface but consequential: it has to look like *the same person*
across every surface they appear on, even when their photo hasn't
loaded — or doesn't exist. The deterministic name → color mapping
solves that without coupling the component to your user database.

---

## Behavior

1. **`src` provided and loads** → the photo is shown.
2. **`src` missing or fails to load** → initials derived from `name`
   appear on a colored tile.
3. The tile color is **deterministically hashed from `name`**, so the
   same person always picks the same color across the app and across
   re-renders.
4. Override the color signal manually with `appearance` (named preset)
   or `palette` (custom set), or override the underlying CSS variables
   for full per-instance control.

## Same person, same color

The default `appearance="auto"` hashes `name` deterministically (a
stable djb2 over the string) and picks an entry from the palette. Same
name → same color, every render, every surface. Three avatars with the
same name always pick the same tile:

## Forcing a specific color

Sometimes you want to **override** the hashed color — for example, to
mark the current user, to flag a special role, or to keep all members
of one squad on the same hue.

<Canvas of={AvatarStories.Appearances} />

| `appearance` | Effect                                                  |
| ------------ | ------------------------------------------------------- |
| `auto`       | Hash `name` to pick a palette entry. **Default.**       |
| `neutral`    | Muted grey surface — no color signal.                   |
| `blue`       | Blue tile.                                              |
| `green`      | Green tile.                                             |
| `red`        | Red tile.                                               |
| `yellow`     | Yellow tile.                                            |
| `purple`     | Purple tile.                                            |

<Source
  language="tsx"
  code={`<Avatar name="Wasim Khan" appearance="purple" />`}
/>

## Custom palettes

Pass `palette` to swap the default 8-color set with your own — useful
for matching a brand palette or a specific accessibility constraint.
The hash-by-name math runs over the supplied array, so the length and
order of the palette determines which color a given name picks.

<Canvas of={AvatarStories.CustomPalette} />

<Source
  language="tsx"
  code={`import { Avatar, type AvatarPaletteEntry } from "@catylast/primitives";

const brandPalette: AvatarPaletteEntry[] = [
  { bg: "#0F62FE", color: "#FFFFFF" },
  { bg: "#F1C21B", color: "#1F2933" },
  { bg: "#198038", color: "#FFFFFF" },
  { bg: "#D12771", color: "#FFFFFF" },
  { bg: "#491D8B", color: "#FFFFFF" },
];

<Avatar name="Wasim Khan" palette={brandPalette} />`}
/>

### Extending the default palette

If you want the Catylast default palette plus a few brand tones, spread
`DEFAULT_AVATAR_PALETTE` and append your own entries:

<Source
  language="tsx"
  code={`import {
  Avatar,
  DEFAULT_AVATAR_PALETTE,
  type AvatarPaletteEntry,
} from "@catylast/primitives";

const extended: AvatarPaletteEntry[] = [
  ...DEFAULT_AVATAR_PALETTE,
  { bg: "#0EA5A0", color: "#FFFFFF" }, // teal
  { bg: "#FB7185", color: "#1F2933" }, // soft pink
  { bg: "#F97316", color: "#FFFFFF" }, // orange
];

<Avatar name="Maya Patel" palette={extended} />`}
/>

## CSS variable escape hatch

For one-off overrides — a personal accent color, a special role, a
holiday theme, gradient backgrounds, square corners — set the
underlying CSS variables directly via `style`. Inline styles win over
both the `appearance` preset and the auto-mode inline color, so
consumers always have final say.

<Canvas of={AvatarStories.CssVariableEscapeHatch} />

### Variables published

| Variable                  | Default                | Purpose                       |
| ------------------------- | ---------------------- | ----------------------------- |
| `--avatar-size`           | per size               | Box width / height.           |
| `--avatar-font-size`      | per size               | Initials font size.           |
| `--avatar-font-weight`    | `semibold`             | Initials font weight.         |
| `--avatar-radius`         | `radius.full`          | Corner radius.                |
| `--avatar-bg`             | per appearance / hash  | Tile background color.        |
| `--avatar-color`          | per appearance / hash  | Initials text color.          |
| `--avatar-border-color`   | `border.subtle` / `transparent` | Outer border color.   |
| `--avatar-border-width`   | `1px`                  | Outer border width.           |

## Accessibility

- The avatar root has `role="img"` and `aria-label={name ?? "User"}`,
  so a screen reader announces the image meaningfully whether or not
  the photo loaded.
- The `<img>` itself has `alt=""` because the surrounding label already
  carries the meaning — preventing duplicate announcements.
- Initials and tile colors are decorative — never relied on as the
  only signal of identity. Pair the avatar with the author name in
  text whenever you need certainty.

---

## Best practices

- **Always pass `name`.** Initials and the deterministic color both
  require it; without a name, the avatar shows a generic `?` on a
  neutral tile.
- **Prefer `appearance="auto"` for people**, where the visual variety
  helps scannability. Use `neutral` for *system* / *bot* / *deleted*
  identities so they read as different from real users.
- **Don't hash sensitive input.** `name` is fine — it's already
  visible in the UI. Don't pass an email or user ID into a name slot
  just to colourise the tile; that leaks data into the rendered HTML.
- **Cap palettes around 8–12 entries.** Larger palettes diminish
  returns (humans don't reliably distinguish 20 hues at small avatar
  sizes) and increase the chance that a person's chosen color clashes
  with the surrounding UI.
- **Match the radius to the surface.** `radius.full` (round) is the
  default. For card-style identity tiles or table-row avatars, square
  or `radius.md` reads cleaner.
