<Subtitle>
  The input + calendar combo Catylast uses everywhere a user picks a
  single day — due dates, scheduling, deadlines, filter ranges. The
  input accepts free typing in a configurable format, and the trailing
  calendar icon opens a popover with the full Calendar primitive.
</Subtitle>

`DatePicker` ships from **`@catylast/primitives`**. It composes
the underlying `Calendar` primitive inside a Radix Popover with a
custom input trigger. Locale-aware parsing and formatting use
`Intl.DateTimeFormat`, so the same component works for any region
without a date library.

The `value` exchanged with consumers is always a canonical ISO
`YYYY-MM-DD` string. Display formats are purely a presentation concern
and never round-trip lossy: switching `dateFormat` from `YYYY-MM-DD`
to `DD/MM/YYYY` doesn't change the data, only the input's display.

## Quickstart

The minimum viable picker — uncontrolled, no value, default
`YYYY-MM-DD` format:

<Canvas of={DatePickerStories.Default} />

<Source
  language="tsx"
  code={`import { DatePicker } from "@catylast/primitives";

<DatePicker onChange={(iso) => console.log(iso)} />;`}
/>

Click the calendar icon to open the popover, or just start typing — the
parser commits the value on blur.

---

## Controlled vs uncontrolled

Pass `value` + `onChange` for fully-controlled behaviour, or use
`defaultValue` for uncontrolled. The `onChange` callback always
receives a canonical ISO `YYYY-MM-DD` string (or `""` when cleared).

<Canvas of={DatePickerStories.Controlled} />

<Source
  language="tsx"
  code={`// Uncontrolled
<DatePicker defaultValue="2026-05-12" />

// Controlled
const [iso, setIso] = useState("2026-05-12");
<DatePicker value={iso} onChange={setIso} />`}
/>

## Custom date format

`dateFormat` controls the display + parse pattern. Tokens:

| Token | Meaning                             |
| ----- | ----------------------------------- |
| `YYYY` | 4-digit year                       |
| `MM`   | 2-digit zero-padded month          |
| `M`    | Month, no leading zero             |
| `DD`   | 2-digit zero-padded day            |
| `D`    | Day, no leading zero               |

Free typing is parsed against the same pattern the user sees, so what
they type is what they get.

<Canvas of={DatePickerStories.CustomFormat} />

<Source
  language="tsx"
  code={`<DatePicker dateFormat="DD/MM/YYYY" />          // 12/05/2026
<DatePicker dateFormat="M/D/YYYY" />            // 5/12/2026
<DatePicker dateFormat="YYYY/MM/DD" />          // 2026/05/12`}
/>

For richer parsing — month names, fuzzy strings — pass
`parseInputValue(input, dateFormat)` and `formatDisplayLabel(iso,
dateFormat)` to plug in your own logic.

## Range constraints

`minDate` / `maxDate` clamp the entire selectable range. Out-of-range
days disable in the calendar; the prev / next month arrows disable
themselves when navigation would leave the range.

<Canvas of={DatePickerStories.MinMax} />

<Source
  language="tsx"
  code={`<DatePicker minDate="2026-05-04" maxDate="2026-05-22" />`}
/>

### Disabled dates

Pass an array of ISO strings to `disabledDates` for non-contiguous
gaps — booked dates, weekends, holidays. Disabled days render with a
strikethrough and don't fire `onChange`.

<Canvas of={DatePickerStories.DisabledDates} />

<Source
  language="tsx"
  code={`<DatePicker
  disabledDates={[
    "2026-05-02", "2026-05-03",
    "2026-05-09", "2026-05-10",
    // ... weekends
  ]}
/>`}
/>

| --------------------------------------- | ------------------------ | ------------------------------------------ |

| `--datepicker-radius`                   | `radius.sm`              | Trigger corner radius.                     |
| `--datepicker-padding-x` / `-y`         | per size                 | Trigger padding.                           |
| `--datepicker-min-height`               | per size                 | Trigger minimum height.                    |
| `--datepicker-width`                    | `240px`                  | Trigger width.                             |

## Form submission

The picker renders a real `<input type="text">` underneath the trigger,
so it participates in standard HTML form submission via the `name`
attribute:

<Source
  language="tsx"
  code={`<form action="/save" method="post">
  <DatePicker name="due_date" defaultValue="2026-05-12" />
  <button type="submit">Save</button>
</form>`}
/>

The submitted value is whatever the user typed (or last picked),
formatted by `dateFormat`. If you need the canonical ISO on the
backend, parse it server-side or hold the `value` in React state and
submit it via `fetch`.

## Accessibility

- The input is a real `<input type="text">` with `role="combobox"`,
  `aria-haspopup="dialog"`, and `aria-expanded` wired up.
- `aria-controls` points at the popover so screen readers can find the
  associated calendar grid.
- `aria-invalid` reflects `isInvalid`; pair with descriptive text
  (e.g. an `aria-describedby` error message).
- The calendar inside the popover is the same `Calendar` primitive
  documented under **Forms / Calendar** — the full W3C grid pattern,
  arrow-key navigation, `aria-selected`, `aria-current="date"` for
  today, locale-aware long `aria-label` on each day.
- The clear (`×`) button has `aria-label="Clear date"` and the trigger
  icon has `aria-label="Open calendar"`.
- Pressing `↓` in the input opens the popover so keyboard-only users
  don't need to tab to the icon button.

## Best practices

- **Always provide an `aria-label`** when the picker has no visible
  label. Screen-reader users hear the placeholder otherwise, which
  rarely conveys meaning.
- **Constrain ranges with `minDate` / `maxDate`** in booking flows.
  Otherwise users can navigate to year 5000 by mashing PageDown.
- **Use the same `dateFormat` across the app.** Switching between
  `MM/DD/YYYY` and `DD/MM/YYYY` in different places is a classic
  source of off-by-one bugs in user input.
- **Persist the canonical ISO, not the formatted display.** The
  `onChange` value is always `YYYY-MM-DD` — store that in state and on
  the wire.
- **Reach for `<TimePicker>` (not the calendar) for time-of-day.** The
  two compose together for full datetime inputs.
