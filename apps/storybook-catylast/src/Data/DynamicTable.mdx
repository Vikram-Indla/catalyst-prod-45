<Subtitle>
  A flexible, theme-aware data table with sorting, multi-row selection,
  hierarchical row expansion, sticky pinned columns, column resizing,
  density modes, and column visibility — built on TanStack Table v8 with
  fully owned markup, styling, and accessibility wiring.

`DynamicTable` is the workhorse for every tabular surface in Catylast: issue
lists, work-item trees, settings tables, audit logs, dashboards. The minimum
signature is two props: `columns` and `data`. **Sorting, column resizing,
and multi-row selection (with the floating action strap) are on by default**
— pass `enableSelection={false}` to suppress selection on read-only surfaces.
Expansion, column visibility, and pinning are separate opt-in flags.

Cells aren't strings — they're render functions. Compose `Badge`, `Avatar`,
`Select`, `Combobox`, progress bars, links, or any custom component. The
table renders the *machine* (DOM, sticky math, state); you provide the
*content*.

## Quickstart

The minimum viable table — pass `columns` and `data`:

<Canvas of={DynamicTableStories.Default} />

<Source
  language="tsx"
  code={`import { DynamicTable } from "@catylast/dynamic-table";

const columns = [
  { accessorKey: "summary", header: "Summary" },
  { accessorKey: "status", header: "Status" },
  { accessorKey: "assignee", header: "Assignee" },
];

<DynamicTable columns={columns} data={issues} />;`}
/>

Sorting and column resizing are enabled by default. Click any header to sort,
drag a header's right edge to resize.

## When to use

| Pattern                              | When to reach for it                                                                |
| ------------------------------------ | ----------------------------------------------------------------------------------- |
| **Issue lists & work-item trees**    | Tabular work data that benefits from sorting, selection, or hierarchy.              |
| **Settings tables with inline edit** | Status / assignee / role / scope where editing in-row avoids navigating away.       |
| **Audit logs & event feeds**         | Many columns where users need to pin the timestamp / actor and scroll the rest.     |
| **Dashboards**                       | Structured rows of measurable items — projects, releases, deployments, builds.      |

### When **not** to use

- **Pure visual layouts** — use `Box`, `Stack`, `Inline`, or `Grid`.
- **Key-value pairs about one entity** — use `DescriptionList`.
- **Charts and time-series** — use a chart component.
- **Lists of richly-laid-out items** — use a `Card` grid.
- **Truly enormous datasets** (≥ a few thousand rows on screen at once).
  v0.1 doesn't ship row virtualization yet — paginate upstream until v0.2.

## Anatomy

```text
┌──────────────────────────────────────────────────────────────────┐
│ ┌──────────────────────────────────┐  ┌────────┐  ┌────────────┐ │  ← toolbar
│ │ search / filters (toolbarLeft)   │  │ filter │  │  ⚙ columns │ │
│ └──────────────────────────────────┘  └────────┘  └────────────┘ │
├──────────────────────────────────────────────────────────────────┤
│ ☐  Title  ↑    Status      Priority   Assignee   Points          │  ← header
├──────────────────────────────────────────────────────────────────┤
│ ☐  ▾ ⚡ CAT-1  Issue       In progr…  High       👤 Wasim    21  │  ← row
│ ☐    📖 CAT-2    ↳ Child   Done       Medium     👤 Alex      5  │  ← child
│ ☐  ⚡ CAT-6  Another epic  To do      Low        👤 Maya      8  │
├──────────────────────────────────────────────────────────────────┤
│ +  Create epic                                                   │  ← creator
└──────────────────────────────────────────────────────────────────┘
       ↑                                                  ↑
   pinned column                                  scrolls horizontally
```

- **Toolbar (optional)** — appears above the table when
  `enableColumnVisibility` is on or `toolbarLeft` / `toolbarRight` slots
  are provided.
- **Header row** — sticky to the top of the scroll area. Sortable headers
  click to sort; resize handles appear on hover at the right edge.
- **Body rows** — depth-aware (children indent 24px per level when
  expansion is enabled). Hover row actions fade in when
  `renderRowActions` is provided.
- **Creator footer (optional)** — sticky to the bottom-left of the body.
  Pair `createLabel` + `onCreate` for the default button, or use
  `renderCreator` for custom content.
- **Pinned columns** — anchored to the left during horizontal scroll, with
  a 1px right-edge divider on the last pinned column.

## Sorting

Enabled by default. Clicking a header cycles **unsorted → ascending →
descending → unsorted**. The chevron indicator appears next to the header
label. Disable per column with `enableSorting: false` on its `ColumnDef`,
or globally with `enableSorting={false}` on the table.

## Pinned columns

Pin one or more columns to the left so they stay visible during horizontal
scroll. The last pinned column gets a 1px divider that separates the
sticky and scrolling regions.

<Canvas of={DynamicTableStories.PinnedColumns} />

<Source
  language="tsx"
  code={`<DynamicTable
  columns={columns}
  data={issues}
  pinnedColumns={["summary", "status"]}
/>`}
/>

> **Tip:** pin the column that identifies the row — usually the title or
> key — so users always know which row they're looking at, no matter how
> far they scroll.

## Multi-row selection

**Selection is on by default.** Every `DynamicTable` ships with a leading
checkbox column and a floating dark **action strap** that slides up from
the viewport bottom whenever one or more rows are selected. The header
checkbox toggles all rows; partial selection shows an indeterminate state
announced as `aria-checked="mixed"`. Subscribe to `onSelectionChange` for
the array of selected row originals.

<Canvas of={DynamicTableStories.Selection} />

<Source
  language="tsx"
  code={`// Default — checkbox column + action strap are auto-injected
<DynamicTable
  columns={columns}
  data={issues}
  onSelectionChange={(rows) => setSelected(rows)}
/>`}
/>

### Action strap

The strap is the `ActionStrap` primitive from `@catylast/primitives` —
DynamicTable wires it up for you with five slots: **Select All**, **Edit
fields**, **Change status**, **Watch options**, and **Delete**. The
trailing close icon clears the selection. Each label is paired with a
brand glyph from `@catylast/icons`.

- **Select All** is wired by default — clicking it selects every row in
  the table. When every row is already selected, the same button label
  flips to **Unselect All** and a click clears the selection.
- The other four buttons are visual until you hand them a callback. Pass
  `onEditFields` / `onChangeStatus` / `onWatchOptions` / `onDelete` and
  each handler receives the array of selected row originals.

<Source
  language="tsx"
  code={`<DynamicTable
  columns={columns}
  data={issues}
  onSelectionChange={setSelected}
  onEditFields={(rows) => openBulkEditor(rows)}
  onChangeStatus={(rows) => openStatusPicker(rows)}
  onWatchOptions={(rows) => watchAll(rows)}
  onDelete={(rows) => confirmDelete(rows)}
  // Optional: override "Select All" (e.g. to load every server-side row,
  // not just the visible page)
  onSelectAll={(currentlySelected) => loadAndSelectEverything()}
/>`}
/>

### Opting out

Pass `enableSelection={false}` to suppress both the checkbox column and
the strap entirely. Use this for read-only audit logs, single-select
pickers, or anywhere bulk operations don't apply.

<Source
  language="tsx"
  code={`<DynamicTable
  columns={columns}
  data={issues}
  enableSelection={false}
/>`}
/>

If you want the checkbox column but a custom toolbar instead of the
built-in strap, keep selection on and pass `showSelectionStrap={false}`
— then render your own UI from `onSelectionChange`.

<Source
  language="tsx"
  code={`<DynamicTable
  columns={columns}
  data={issues}
  showSelectionStrap={false}
  onSelectionChange={(rows) => setMyOwnToolbarRows(rows)}
/>`}
/>

## Hierarchical row expansion

Provide `getRowChildren` and set `enableExpansion`. A chevron toggle is
auto-injected into the first column. Clicking the chevron — or pressing
**Enter** / **Space** when focused — toggles expansion. Always pass
`getRowId` so expansion state survives data refreshes.

<Canvas of={DynamicTableStories.RowExpansion} />

<Source
  language="tsx"
  code={`<DynamicTable
  columns={columns}
  data={epics}
  enableExpansion
  getRowChildren={(row) => row.children}
  getRowId={(row) => row.id}
/>`}
/>

## Column visibility

Setting `enableColumnVisibility` adds a *Configure columns* gear menu in
the toolbar. Each menu item is a column; click to toggle, with the menu
staying open so users can flip several at once.

<Canvas of={DynamicTableStories.ColumnVisibility} />

- `defaultColumnVisibility={{ priority: false }}` — hide a column by
  default.
- `enableHiding: false` on a `ColumnDef` — make a column un-hideable.

## Custom cells

Cells are render functions, not strings. Compose any React component into
the cell and the table renders it as-is. The table doesn't know or care
what's in the cell — only the row, header, and pinning math.

<Canvas of={DynamicTableStories.CustomCells} />

### Status badge

<Source
  language="tsx"
  code={`import { Badge } from "@catylast/primitives";

{
  accessorKey: "status",
  header: "Status",
  cell: ({ row }) => {
    const s = row.original.status;
    if (s === "Done") return <Badge variant="success">{s}</Badge>;
    if (s === "In progress") return <Badge variant="primary">{s}</Badge>;
    return <Badge variant="default">{s}</Badge>;
  },
}`}
/>

### Avatar with name

<Source
  language="tsx"
  code={`import { Avatar } from "@catylast/primitives";

{
  accessorKey: "assignee",
  header: "Assignee",
  cell: ({ row }) => {
    const a = row.original.assignee;
    if (!a) return "—";
    return (
      <span style={{ display: "inline-flex", alignItems: "center", gap: 8 }}>
        <Avatar name={a.name} size="xs" />
        {a.name}
      </span>
    );
  },
}`}
/>

## Inline editing

There's no built-in editing mode — you compose it. Render a `Select`,
`Combobox`, or any input inside a cell and dispatch updates from its change
handler. The table doesn't need to know that the cell is editable.

<Canvas of={DynamicTableStories.InlineEditing} />

### Inline status select

<Source
  language="tsx"
  code={`{
  accessorKey: "status",
  cell: ({ row }) => (
    <Select
      value={row.original.status}
      onValueChange={(s) => updateRow(row.original.id, { status: s })}
    >
      <SelectTrigger variant="ghost" hideIcon>
        <Badge>{row.original.status}</Badge>
      </SelectTrigger>
      <SelectContent>
        <SelectItem value="To do">To do</SelectItem>
        <SelectItem value="In progress">In progress</SelectItem>
        <SelectItem value="Done">Done</SelectItem>
      </SelectContent>
    </Select>
  ),
}`}
/>

### Searchable assignee picker

<Source
  language="tsx"
  code={`{
  accessorKey: "assignee",
  cell: ({ row }) => (
    <Combobox
      value={row.original.assignee ?? null}
      options={users}
      getKey={(u) => u.id}
      getLabel={(u) => u.name}
      onSelect={(u) => updateRow(row.original.id, { assignee: u })}
      renderTrigger={(u) => (
        <span style={{ display: "inline-flex", alignItems: "center", gap: 8 }}>
          <Avatar name={u.name} size="xs" />
          {u.name}
        </span>
      )}
    />
  ),
}`}
/>

> **Use ghost variants** on `Select` / `Combobox` triggers so the cell
> looks like a regular cell with a subtle hover state rather than a
> pre-styled input.

## Density

Density flips CSS variables; cells with multi-line content (e.g., title +
progress bar) grow above the minimum naturally.

| Density        | Min row height | Cell padding (Y / X) | When to reach for it                                  |
| -------------- | -------------- | -------------------- | ----------------------------------------------------- |
| `compact`      | 28px           | 4px / 8px            | Dense scanning — release notes, log entries.          |
| `standard`     | 36px (default) | 8px / 12px           | Most cases.                                           |
| `comfortable`  | 44px           | 12px / 16px          | Multi-line content, rich controls, padded inputs.     |

## Hover row actions

Provide `renderRowActions(row)` to inject `+` and `…` buttons into the
first column on row hover. Clicks inside row actions do not trigger
`onRowClick`. Long titles auto-truncate to make room when actions appear.

<Source
  language="tsx"
  code={`<DynamicTable
  renderRowActions={(row) => (
    <>
      <IconButton icon="plus" label="Add child" size="sm"
        onClick={() => addChild(row.id)} />
      <Menu>
        <MenuTrigger asChild>
          <IconButton icon="more-horizontal" label="More" size="sm" />
        </MenuTrigger>
        <MenuContent align="end">
          <MenuItem onSelect={() => edit(row.id)}>Edit</MenuItem>
          <MenuItem variant="danger" onSelect={() => del(row.id)}>Delete</MenuItem>
        </MenuContent>
      </Menu>
    </>
  )}
/>`}
/>

## Inline creator footer

The simplest form is `createLabel="Create epic"` + `onCreate={fn}`. For a
type-picker (Epic / Story / Task) wrap a `Menu` yourself in
`renderCreator`. The creator row is sticky to the bottom-left of the body
and stays visible during horizontal scroll.

<Source
  language="tsx"
  code={`<DynamicTable
  renderCreator={() => (
    <Menu>
      <MenuTrigger asChild>
        <Button variant="ghost" size="sm">
          <Icon name="plus" size={14} /> Create
        </Button>
      </MenuTrigger>
      <MenuContent align="start">
        <MenuItem onSelect={() => create("epic")}>Epic</MenuItem>
        <MenuItem onSelect={() => create("story")}>Story</MenuItem>
        <MenuItem onSelect={() => create("task")}>Task</MenuItem>
      </MenuContent>
    </Menu>
  )}
/>`}
/>

## Pagination

Pagination is **on by default** with `pageSize: 20`. The chrome renders
in a fixed footer band **outside the scrolling row region** — even with
hundreds of rows in the table, the next-page button is always visible
without scrolling. When there's only one page worth of data, the chrome
auto-hides so single-page tables don't get a useless `1` button.

Use the **Controls** panel under the canvas below to flip `position`
between `start` / `center` / `end` and to change `pageSize` live.

<Canvas of={DynamicTableStories.Pagination} />

<Controls of={DynamicTableStories.Pagination} />

### Configuration

The `pagination` prop accepts three shapes:

| Value | Behaviour |
| --- | --- |
| **Omitted** (default) | Pagination on. `pageSize: 20`, `position: "center"`, `hideOnSinglePage: true`. |
| `false` | Disabled entirely. Every row renders, no chrome. Use for static reports or export views. |
| Object | Customise any of `pageSize`, `position`, `hideOnSinglePage`, or pass `page` + `onPageChange` for controlled mode. |

<Source
  language="tsx"
  code={`// Default — pageSize 20, centered chrome, hides on single page
<DynamicTable columns={columns} data={rows} />

// Disable entirely — render every row
<DynamicTable columns={columns} data={rows} pagination={false} />

// Custom page size
<DynamicTable
  columns={columns}
  data={rows}
  pagination={{ pageSize: 10 }}
/>

// Position the chrome at start / center / end of the footer
<DynamicTable
  columns={columns}
  data={rows}
  pagination={{ pageSize: 10, position: "start" }}
/>

// Controlled — drive page state from outside (URL-synced, etc.)
<DynamicTable
  columns={columns}
  data={rows}
  pagination={{
    pageSize: 25,
    page,
    onPageChange: setPage,
  }}
/>`}
/>

### Position — start / center / end

Three placements inside the footer band. `center` is the default; use
`start` to pin to the left edge or `end` to pin to the right. The footer
bar itself always sits outside the scroll region, so the chrome stays
reachable regardless of how many rows are in the table.

<Canvas of={DynamicTableStories.PaginationPositions} />

### Scroll behaviour

The container is a vertical flex layout: toolbar (if any), the scrolling
row region, and the pagination footer are three independent rows. The
row region absorbs whatever height is left after the toolbar and footer
claim theirs. This means:

- A bounded-height wrapper (`height: 640px`, a flex parent, etc.) → rows
  scroll inside the bounded region, pagination stays pinned at the
  bottom of the visible container.
- An unbounded wrapper (`height: auto`) → the table renders all rows in
  flow with pagination after them; nothing scrolls.

Either way, the user never has to scroll past hundreds of rows to find
the next-page button.

### Pagination primitive

The chrome itself comes from `@catylast/primitives/Pagination` — a
standalone component you can use anywhere a paginated collection appears
(comment threads, archived issue lists, audit logs, search results). The
DynamicTable just composes it inside its footer slot. See
**Navigation → Pagination** for the primitive's API in isolation.

## Accessibility

- **Keyboard navigation** — every interactive element is reachable via
  Tab. Sortable headers respond to Enter and Space. Expansion toggles are
  real buttons with `aria-label`s that flip between "Expand row" and
  "Collapse row".
- **Selection state** — the header checkbox correctly announces partial
  selection via `aria-checked="mixed"`. Each row's checkbox is associated
  with its row context.
- **Focus rings** — every interactive cell, header, and toggle has a
  visible `focus-visible` outline using the `border.focus` token.
- **Color is not the only signal** — sort direction, status, and selection
  are conveyed via icons, text, and ARIA attributes in addition to color,
  so users with color-vision differences aren't blocked.
- **Cell content is your responsibility** — wrap interactive content
  (links, buttons, inputs) with semantic elements so it stays reachable
  without a mouse. The table's keyboard layer doesn't trap focus inside
  cells.

## Best practices

- **Show fewer than 8 columns** at standard density on a typical 1280px
  viewport. More than that and users will horizontal-scroll constantly —
  pin the most important columns and consider hiding lower-priority ones
  by default with `defaultColumnVisibility`.
- **Pin the row identifier** — usually the title or key — so users always
  know which row they're looking at.
- **Use density purposefully** — compact for dense scanning, standard for
  most cases, comfortable when cells contain multi-line content or
  controls.
- **Lead with text, not icons** — a status pill says "In progress"; don't
  replace the text with just a colored dot.
- **Memoize `columns` and `data`** — re-creating either on every render
  triggers a full TanStack row-model rebuild.
- **Always provide `getRowId`** when data may be refetched. Without it,
  TanStack falls back to row index, which corrupts selection and expansion
  state when rows reorder.

## Content guidelines

- **Column headers** — short noun phrases. Title-case the first word only.
  Avoid abbreviations unless universally understood (Avg, # / count).
- **Action labels** — action verb first ("Edit", "Delete", "Move to…").
  Trailing ellipsis means "opens a dialog or submenu". No ellipsis means
  the action happens immediately.
- **Empty state copy** — tell the user *why* the table is empty and *what
  they can do about it*. "No issues match your filters." beats "No data."
- **Loading state** — brief is fine ("Loading…"). For long fetches, use
  `loadingContent` to render a skeleton or a more informative status.
- **Pluralization** — "1 issue selected" / "23 issues selected" — handle
  the plural in your selection-count copy.

## Theme switching

Use the toolbar at the top of the canvas to flip between **Light**,
**Dark**, and **System**. The `data-theme` attribute on `<html>` drives
every token value the table consumes; nothing in this component reads the
theme name directly.
