1. Verified Research Findings

25 claims extracted from 22 sources across 5 search angles. 13 confirmed (3-vote adversarial), 12 killed. Only confirmed findings below.

Asana — View Architecture 3-0 CONFIRMED

Asana provides four standard project views (List, Board/Kanban, Calendar, Timeline) displaying the same underlying data, switchable by any teammate at any time. Dedicated My Tasks view aggregates all user assignments for prioritization.

Catalyst implication: Our 7 views (Task List, Board, Timeline, Priorities, My Tasks, Work Streams, Overview) exceed Asana's model. The same planner_tasks table must power ALL views — no per-view tables.

Sources: asana.com/features, asana.com/inside-asana/manage-workflow-project-views

Asana AI Studio 2-1 CONFIRMED

No-code workflow builder for AI-powered task automations. Reference pattern for CATY AI: surface task recommendations, auto-assign, auto-prioritize, summarize workstream progress.

Source: asana.com/product/ai

Notion — Data Model & Views 3-0 CONFIRMED

Structures tasks as a standalone database with multiple views (Kanban, Timeline, Table, List, Gallery, Calendar) as layout options over the same dataset. Tasks are parentless work items — aligns with Vikram's directive that Catalyst tasks have no parent hierarchy.

Source: notion.com/help/guides/getting-started-with-projects-and-tasks

ADS Dynamic Table 3-0 CONFIRMED

@atlaskit/dynamic-table provides pagination, sorting, and drag-and-drop re-ordering. Catalyst wraps this in JiraTable — the task list MUST reuse JiraTable, not dynamic-table directly.

Source: atlassian.design/components/dynamic-table

ADS Has NO Board/Timeline Components 2-1 CONFIRMED

ADS provides Pragmatic Drag and Drop (@atlaskit/pragmatic-drag-and-drop) as the official DnD library (powers Trello, Jira, Confluence in production). Board and Timeline views must be assembled from this primitive. Catalyst already has PragmaticBoard internally.

Sources: atlassian.design/components/pragmatic-drag-and-drop, github.com/atlassian/pragmatic-drag-and-drop

Per CLAUDE.md "ADOPT CANONICAL COMPONENTS — DO NOT REIMPLEMENT": mount the real component and feed it task data through an adapter. Example:

// Adapter: map planner_tasks row → JiraTable WorkItem shape
function taskToWorkItem(task: PlannerTask): WorkItem {
  return {
    id: task.task_key,           // TSK-001
    issue_key: task.task_key,
    summary: task.title,
    status: task.status,
    status_category: task.status_category,
    assignee_display_name: task.assignee_name,
    priority_name: task.priority,
    issue_type: 'Task',
    created: task.created_at,
    updated: task.updated_at,
    // ... map remaining fields
  };
}
