## 6. tm_defects ‚Üî ph_issues relationship (hybrid model)

There is **no FK** between tm_defects and ph_issues. The bridge is 4-fold:

1. **Issue-type hybrid**: defects/incidents at the reporting layer = `ph_issues` rows with issue_type 'QA Bug' (788) / 'Production Incident' (152) UNION tm_defects (locked decision D-005, memory testhub-report-revamp; implemented in `src/pages/testhub/reports/lab/reportData.ts:273` "defect selectors (tm_defects)" + ph_issues selectors).
2. **parent_key**: `tm_defects.parent_key TEXT` ‚Üí matches `ph_issues.issue_key` (`20260628130000_tm_defects_adf_parent.sql:9`; consumed at `reportData.ts:63`: "tm_defects.parent_key ‚Üí ph_issues.issue_key").
3. **Creation flow**: TestHub Create opens canonical CreateStoryModal with work type QA Bug; the `isDefect` branch writes to tm_defects, not ph_issues (memory `defect-creation-canonical-qabug.md`).
4. **Workflow bridge**: `tm_defects.workflow_status_key` maps to canonical `ph_wf_*` status keys via Defect SDLC workflow v1 (new‚Üítriage‚Üíaccepted‚Üíassigned_for_fix‚Üí‚Ä¶; rejected/duplicate/deferred exceptions) seeded in `20260628290000_ph_wf_defect_adapter.sql` ‚Äî advisory only, enum `status` never widened.

Other cross-family links: tm_test_cases.linked_story_key‚Üíph_issues.issue_key (text, `20260702200001`); tm_requirement_links.requirement_id‚Üíph_issues.id (uuid, app-resolved, no FK); tm_test_cases/tm_test_cycles/tm_test_plans/tm_defects.sprint_id‚Üí**ph_jira_sprints(id)** (real FK per types.ts); release_id‚Üí**releases(id)** (legacy releases table, NOT rh_releases).
