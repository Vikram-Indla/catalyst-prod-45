# Decisions

- Removed the "Plan" hub entirely (was already flagged `deprecated: true` in HubSwitcher.tsx). No replacement UI.
- Kept `/planhub*` → `/tasks/overview` redirect in FullAppRoutes.tsx so old bookmarks/links don't 404.
- Did NOT drop any `planhub_*` Supabase tables/enums or regenerate `integrations/supabase/types.ts` — DB schema changes are out of scope for a UI-module retirement and carry migration risk; the generated types file simply reflects the (unused) real schema.
- Did NOT touch `src/constants/notificationEvents.ts` / `TriggerRow.tsx` / `TriggerSearch.tsx` / `NotificationTriggers.tsx` PlanHub-labeled entries. On inspection, the "PLAN_HUB_EVENTS" block (task_created, task_assigned, workstream_created, allocation_created, etc.) describes Tasks/Workstreams notification events, not the deleted capacity/budget-planner Plan Hub UI — ambiguous overlap flagged by discovery agent (moduleKey `planner` is shared between Tasks and the old Plan hub). Removing these risked regressing live Tasks notifications. Left untouched.
- Did NOT touch `src/registry/usage-map.generated.ts` (auto-generated by scripts/scan-components.ts; regenerates on next run).
- Left star/favorites generic-logic tests (`StarredHubList.test.tsx`, `useMigrateLegacyFavorites.test.ts`, `starType.test.ts`) as-is — they use `/planhub/capacity` only as example fixture data for generic path-matching logic, not gated on the module existing.
- Fixed the login page marketing copy ("One platform. Eight connected modules." → "...Seven...") and removed the Plan tile from the MODULES carousel (en + ar) since it would otherwise advertise a nonexistent feature.
